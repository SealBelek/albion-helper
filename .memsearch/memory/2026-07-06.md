Previously, NATS goroutines, TUI refresh, cleanup, and API goroutines all competed for separate connections, each creating its own TLS/mmap overhead.

- **`PRAGMA cache_size=-8000`**: Limits SQLite's page cache to 8MB instead of the default (~2GB). Prevents unbounded memory growth from SQLite's internal cache.

- **`PRAGMA mmap_size=0`**: Disables memory-mapped I/O. On pure-Go `modernc.org/sqlite`, `mmap` allocations go through `modernc.org/memory.(*Allocator).mmap` which **never returns memory to the OS**. This was showing up as 4.5MB per allocation in profiles. Disabling mmap forces all I/O through read/write syscalls instead.

- **Batched cleanup DELETE**: Changed from `DELETE FROM marketorders WHERE created_at <= ...` (single unbounded transaction) to a loop that deletes 1000 rows at a time. This prevents large dead-space accumulation and long-held write locks.

### `internal/nats/subscriber.go` — Batched order inserts

Instead of one `INSERT` per NATS message (causing 100+ individual transactions/second), orders are now buffered and flushed in batches:

- Orders accumulate in `[]orderEntry` slice protected by `sync.Mutex`
- A `flushLoop` goroutine runs every 100ms or when 50 orders accumulate (`orderBatchSize`)
- Each flush does a single `BEGIN TRANSACTION` → prepared statement → batch of `Exec` calls → `COMMIT`
- This reduces transaction overhead from ~100/sec to ~10/sec, and TLS/mmap allocations proportionally
- On `Stop()`, remaining orders are flushed beforeexit

### `cmd/main.go` — Force GC before profiling

Added `runtime.GC()` before `runtime.ReadMemStats()` in the debug profiling loop. This gives more accurate heap profiles by forcing a GC cycle before snapshotting, which reduces noise from dead-but-not-yet-collected objects in the profile data.
